1 The Big Picture  — five zones, one flow

How to read the diagram

left → right = the journey of one AI request
  • Left side: Who sends requests — agents, workflows, apps. These are your callers, not TokenIQ components.
  • Middle (teal box): The control plane — the 8 stages every request passes through.
  • Right side: The model providers that actually answer the call — OpenAI, Anthropic, Bedrock, internal.
  • Below middle: The data stores and learning loop that make the system smarter over time.
SAY: "Think of the middle box as your org's AI traffic controller. Nothing reaches a model without going through it."

The five zones at a glance

what each zone does in plain English
  • ① AI Producers — the callers. Coding agents, bots, apps. They just make normal API calls.
  • ② Control Plane — the 8-stage pipeline. This is TokenIQ's core.
  • ③ Model Endpoints — the providers. The control plane decides which one gets the call.
  • ④ State & Telemetry — memory. Redis, vector store, Postgres. Not live yet — production target.
  • ⑤ Learning Loop — the compounding moat. Gets smarter as usage grows. Roadmap.

1 Gateway Receives the request. Accepts the same format as OpenAI's API so callers need zero code changes. The front door. Looks identical to every door the caller has used before. "Callers don't know they're talking to TokenIQ. They just call an API."
2 Meter + Tag Stamps the request with team, run_id, and correlation_id so every call can be traced back to a team, a ticket, and a workflow run. The barcode scanner at a warehouse door. Every item gets labelled before it moves anywhere. "This is how you get chargeback. Without the tag, you can't attribute cost to anyone."
3 Semantic Cache Checks if this question — or one very close to it — was already answered. Exact match first, then fuzzy/semantic match. If yes: returns the answer instantly, costs zero tokens. Checking your notes before calling the expert again. If you wrote down the answer last time, you don't pay for another call. "About 1 in 3 calls hit the cache in our simulation. Each one is 100% savings — no model, no cost."
4 Budget Governor Two checks: (a) Has this exact prompt been sent too many times? (loop detection). (b) Has this run burned through its token budget? If either trips, the call is killed before it reaches a model. A pre-paid card with a fraud detection rule. Spend freely up to the limit; get automatically blocked if you look like you're stuck in a loop. "Agents can get stuck. Without this, a looping agent can burn thousands of dollars overnight. This makes that physically impossible."
5 Model Router Looks at the task type and picks the cheapest model tier that's capable enough to do it well. Simple tasks go to small/cheap models. Only complex tasks go to frontier. An expense policy — junior staff fly economy, not business. But if the task genuinely needs business class, it gets it. "Most tasks don't need GPT-4. Classify, retrieve, summarize — cheap models handle these fine. Only codegen needs the big model."
6 Compression Adapter Shrinks the prompt before it hits the model — strips redundant context, whitespace, and boilerplate. You're billed for fewer input tokens. Summarising a 10-page brief into 2 pages before sending it to a consultant who charges by the page. "For RAG and code review tasks, up to 70–85% of the input is compressible. We strip it before billing."
7 Model Dispatch Sends the compressed request to the selected provider (OpenAI, Anthropic, Bedrock, internal LLM) via LiteLLM. This is the only stage that touches the outside world. The actual phone call to the expert. Everything before this was preparation; this is the one moment money is spent. "In the PoC this is simulated. In production, LiteLLM handles the provider routing — one interface, any model."
8 Quality Gate Checks if the model's response meets the quality bar for this task. If it fails, it automatically escalates one tier and retries — without involving a human. If it passes, the response goes back to the caller. A QA check at the end of a production line. Reject the part, send it to the next station for rework. Only ship what passes. "The system self-corrects. A cheap model that fails escalates automatically. The caller gets a good answer either way."
The fast path shortcut: If stage ③ (cache) gets a hit, the request skips stages ④ through ⑧ entirely. No budget check, no routing, no model call, no billing. Cache hits are the most valuable optimisation on the board.

3 The Two Edges  — who sends requests, who answers them

① Left edge — AI Producers

who sends the calls
  • Coding Agents (Claude Code, Cursor, Codex) — automated engineers opening PRs, writing tests, generating changelogs.
  • Agentic Workflows (LangGraph, internal) — multi-step orchestration pipelines. One "job" can be dozens of LLM calls.
  • RAG / Copilots — knowledge retrieval bots, Q&A assistants. High volume, mostly cheap tasks.
  • Enterprise Apps (ServiceNow, Teams, internal portals) — humans triggering AI via forms and bots.
SAY: "These are your org's callers. They all point at one URL — TokenIQ. That's the whole integration."

③ Right edge — Model Endpoints

who answers the calls
  • OpenAI — GPT-4o, o1, etc. The default frontier option for most teams today.
  • Anthropic — Claude models. Strong on reasoning and long-context tasks.
  • AWS Bedrock — managed multi-model API. Preferred when data must stay in AWS tenancy.
  • Internal LLMs — private fine-tuned models. Used for sensitive data that can't leave the network.
SAY: "The router picks the provider. Callers never know which one answered — and they don't need to."

4 Production State & Telemetry  — what the three stores do

Important context: In the PoC today, all three stores are replaced by in-memory dicts. The control plane logic is identical — the stores are just where the state lives in production. Swapping in Redis + Postgres requires no changes to the gateway or control plane code.

What the stores hold

each store has a specific, non-overlapping job
  • Redis — live counters. How many tokens has run-X burned? Is this team over rate limit? Fast, volatile, sub-millisecond reads.
  • Vector Store — cache embeddings. "Has a similar prompt been answered before?" Semantic similarity search at millisecond speed.
  • Postgres — permanent record. Every call, every cost, every quality score. The source of truth for FinOps and audit.

Why keep them separate

each store is optimised for its workload
  • Redis is fast and ephemeral — perfect for hot-path counters that reset each run.
  • Vector store is optimised for nearest-neighbour search — a relational DB would be too slow for semantic cache lookups.
  • Postgres is durable and queryable — finance needs SQL joins across teams, dates, and task types.
SAY: "One gateway writes to all three. Telemetry is fire-and-forget — it never slows down the hot path."

5 Learning & Economics Loop  — how it compounds over time

Why this is the moat

the compounding advantage competitors can't copy
  • Day 1: routing is rules-based — task type maps to a fixed tier. Good, but static.
  • Day 90: the Router Trainer has observed which model actually passed quality for which task at which cost. Routing becomes evidence-based.
  • Day 365: telemetry covers seasonal patterns, team differences, model drift. The router self-corrects as providers change.
  • The more calls go through, the better the routing gets. A competitor starting fresh starts cold.
SAY: "The data flywheel is the defensible part. You can copy the rules. You can't copy two years of call history."

The four loop components

roadmap — not running today
  • Router Trainer — reads Postgres telemetry, identifies which model had the best cost/quality ratio per task fingerprint, updates routing policy.
  • Outcome Linker — joins correlation_id to real SDLC events: did the PR merge? Did the test pass? Did the ticket close?
  • FinOps Dashboard — showback (here's what your team spent) and chargeback (here's the invoice). Connects to finance tooling.
  • Executive KPIs — cost per resolved ticket, quality trend, risk avoided. The numbers the CPTO and CFO actually care about.

When asked about the loop (say this):
  • None of these four components are running today. They're on the roadmap and depend on Postgres telemetry being wired up first.
  • The correlation_id field is already emitted by the gateway — that's the hook the Outcome Linker needs. The plumbing is in place; the consumer isn't built yet.
  • Right framing: "The PoC proves the hot path works. The loop is what turns it from a cost tool into a business intelligence platform."

6 Hard Questions  — how to answer them

QuestionWhat to say
"Why 8 stages? Isn't that over-engineered?" Each stage has exactly one job and can be turned off independently. Remove the cache → routing still works. Remove compression → quality gate still works. The stages are composable, not coupled. You don't have to use all 8 on day one.
"What's the latency overhead of passing through 8 stages?" Cache, governor, router, and compression are all in-process — microseconds. The only external calls are the model itself (which you'd make anyway) and the async telemetry write (fire-and-forget, off the hot path). For a 2-second model call, the control plane adds under 5ms.
"What if the router sends a task to a model that can't do it?" That's what stage ⑧ Quality Gate is for. If the response doesn't meet the quality bar, it escalates one tier automatically and retries. The caller always gets a response that passed validation — or an explicit error, not a silent bad answer.
"Does the semantic cache return wrong answers?" Cache hits return the answer from a previous call that passed the quality gate. The threshold is 85% similarity in the PoC — conservative. In production you'd tune this per task type: tighter for codegen, looser for summarization. You can also add a staleness TTL.
"Where does data go? Can my prompts leave the perimeter?" Stages ①–⑥ run inside your trust boundary. The only data that leaves is what you explicitly dispatch to a provider in stage ⑦ — same as today. The cache, governor, compression, and telemetry never touch a third-party service. You can also route data-sensitive tasks to internal LLMs only.
"How is this different from just using LiteLLM?" LiteLLM is a transport layer — it normalises the API across providers. TokenIQ is a control plane — it decides what to send, to whom, at what cost, and whether to send it at all. They're complementary: TokenIQ uses LiteLLM in stage ⑦ for the actual dispatch. LiteLLM has no cache, no governor, no router, and no quality gate.
"What's the correlation_id for?" It's a trace ID that links a model call back to the SDLC event that triggered it — a Jira ticket, a PR, a CI run. Without it, you know you spent $X on AI. With it, you know you spent $X on resolving ticket JIRA-4821 which was merged in 2 hours. That's cost-per-outcome, not just cost-per-token.
"The diagram shows Roadmap items — what's actually built?" Stages ①–⑤ and ⑧ are running PoC today — gateway, meter+tag, semantic cache, budget governor, model router, and quality gate. Stages ⑥–⑦ (compression adapter, model dispatch) are simulated in the PoC and are PoC/Production Target. The data stores (Redis, vector, Postgres) and the full learning loop (Router Trainer, Outcome Linker, FinOps) are on the roadmap.
