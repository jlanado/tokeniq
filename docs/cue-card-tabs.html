1 Agent Simulator  — what you're looking at

The request stream table

rows appear live as the agent works through steps
  • Each row is one LLM call — one step in the workflow.
  • The Decision column shows what the control plane did to that call.
  • Tokens and Cost show what was actually spent after optimisation.
SAY: "Think of this as the audit log. Every AI call is visible, priced, and explained."

The KPI strip (top cards)

updates after the run completes
  • Tokens Used — compressed tokens billed; "naive: N" below it is the all-frontier, no-cache baseline.
  • Cache Hit Rate — share of calls that cost $0. ~31% on JIRA-4821 (5 of 16 steps).
  • Routing Savings — cost reduction vs all-frontier. ~84% on JIRA-4821 because mid ($3) handles every task naive sent to frontier ($15).
  • Quality Score — modelled score; 0.89 vs 0.93 naive = ~4% drop, quality held.
  • Run Cost — dollar cost of this run; "naive: $N" shown below it.
  • Compression — how much input context was stripped before the model call. ~38% on JIRA-4821.
SAY: "93% cost reduction. Quality barely moved. That's the whole story."

Task typeWhat it isModel tierPlain English
classify Label or triage a task — is this a bug, a risk, a priority? small "Sorting mail. Any small model can do it."
retrieve Fetch and return a chunk of code or config — no reasoning required. small "Looking something up in a filing cabinet. Simple pattern match."
summarize Condense incidents, test results, or context into a short digest. small "Summarising meeting notes. High volume, low stakes."
write_tests Generate unit tests for a specific behaviour or edge case. mid "Needs reasoning about edge cases, but not full system design."
review Assess a proposed fix or PR for correctness, side effects, and risks. mid "A senior code review. Needs judgement — worth paying mid-tier for."
codegen Generate a correct, production-ready code change or changelog entry. mid "Writing the actual fix. Needs reasoning, so it routes to mid — frontier is never called because mid (cap 0.88) already clears the codegen bar (0.85)."
The routing story in one sentence: Out of 16 steps, frontier is called zero times. classify / retrieve / summarize → small. write_tests / review / codegen → mid (mid cap 0.88 ≥ codegen req 0.85). 5 steps hit cache and cost $0. That stack is where the 93% cut comes from.
Honest caveat (say this if asked):
  • This is a scripted simulation, not live LLM calls. The workflow steps are deterministic — same seed, same result every time.
  • Quality scores are modelled probabilities, not real evals. Production uses an LLM judge or human graders.
  • Cache matching here is token-similarity. Production uses vector embeddings for semantic matching.

3 Policies  — what you're looking at

What policies are

the control surface — who can spend what, on what
  • Think of this as IAM for AI spend. Same concept as cloud resource policies, applied to model calls.
  • Each policy is a rule the control plane enforces before a call reaches a model.
  • Policies are scoped: per-team, per-agent, per-task-type, or org-wide.
SAY: "You wouldn't give every developer an unlimited AWS budget. Same principle here."

The four policy types

what each category governs
  • Rate limits — max calls per minute/hour. Prevents thundering herd.
  • Budget caps — max $ per run, per day, per team. Hard stop, not a warning.
  • Routing rules — which model tiers a team/agent is allowed to reach.
  • Kill thresholds — loop detection: max identical calls, max token burn per run.
SAY: "Finance approves the budgets. Platform sets the caps. Developers just ship."
Policy typePlain English analogyWhat to say
Rate limit Speed limit on a road. The agent can go, just not infinitely fast. "One runaway agent can't starve every other team's requests."
Budget cap A pre-paid card. The agent spends freely until it hits the cap — then it stops, not you. "We gave the agent a $5 budget. It physically cannot spend $500."
Routing rule An expense policy — juniors fly economy, execs fly business. "Dev teams get small + mid. Only approved agents can touch frontier."
Kill threshold A circuit breaker. Three identical API calls in a row trips it — same as electrical systems. "Loops get killed automatically. No pager, no alert fatigue."
Honest caveat (say this if asked):
  • The policy table is display only in this demo — values are illustrative, not live config.
  • Enforcement in the simulation is hard-coded. In production, policies would be stored in a DB and evaluated at request time.
  • Production roadmap: policies as YAML/API config → per-org overrides → policy-as-code with git-backed audit trail.

QuestionWhat to say
"Is this real data?" No — this is a reference architecture simulator. The numbers come from a deterministic model seeded with realistic cost profiles. The pattern is real; the exact figures are illustrative. That's intentional — it lets us show the concept without needing live API keys.
"How do the fleet numbers hold up for our workload?" They're directional. Your actual savings depend on how repetitive your tasks are (cache rate) and your task complexity mix (routing). We'd instrument your real traffic for 2 weeks to get a proper baseline.
"What stops the model from ignoring the policies?" The control plane sits in front of the model, not next to it. The request never reaches the model if the policy says no. It's a proxy, not a prompt instruction — you can't jailbreak it.
"What if we already have LiteLLM / a gateway?" TokenIQ sits on top of — or alongside — existing gateways. The router, cache, governor, and compressor are the logic layer. The transport layer is pluggable.
"How do you measure quality without human review?" In this demo, quality is modelled. In production, you'd back it with an LLM judge, task-specific rubrics, or golden-set evals. The governor can be configured to kill a call if quality drops below a threshold.
"Can it route across different LLM providers — OpenAI, Anthropic, Gemini?" Not in this demo — the router picks a tier (small/mid/frontier) but the provider is fixed. The next version makes the provider pluggable too. The key dependency is observability: once every call logs task type, model, cost, latency, and a quality score, the router can query that history and ask "which provider has the best cost/quality ratio for this task right now?" — and route there. Observability is the training data for intelligent routing.
"What's the difference between rules-based and intelligent routing?" Rules-based is what you see today — task type maps to a fixed model tier, hardcoded, no feedback. Intelligent routing learns from every call: it tracks which model actually performed best for which task at what cost, and routes based on that evidence. Same pattern as how Netflix routes traffic — not by rule, but by what the data says works best right now.
