2 What each button does

▶ Run agent — resolve JIRA-4821

the "good agent" / optimization story
  • One agent runs a real 16-step workflow to fix a bug ticket: triage → fetch code → summarize incidents → generate fix → review → write tests → changelog.
  • Watch each step route to the cheapest model that passes: small or mid. frontier never appears — mid (cap 0.88) already clears the codegen bar (0.85).
  • Repeated calls return CACHE HIT for free. 5 of 16 steps hit cache (steps 3, 6, 9, 12, 15).
  • If a model can't pass the quality bar it upgrades one tier automatically: ESCALATED.
  • A small purple ⊟ after the badge means the prompt was compressed before the call — it's an inline marker, not a separate badge.
SAY: "Same work, ~93% cheaper, quality basically unchanged."

⚠ Run buggy agent (retry loop)

the "bad agent" / governance story
  • 5 normal steps, then the same "regenerate fix" call repeated 6 times — a broken agent stuck in a loop.
  • The governor counts identical calls. On the 4th repeat it fires KILLED — "loop detected".
  • The run stops mid-stream. No more tokens burned.
  • Also enforces a separate 800K-token-per-run budget cap.
SAY: "A runaway agent can't bankrupt you. The kill switch caps the blast radius automatically."

5 If someone pushes back (honest caveats)

This is a reference architecture / simulator, not production. Say so up front — it builds credibility.
  • Quality (Q) is modeled, not measured. In production you'd back it with real evals — graded tasks, human review, or an LLM judge — before claiming "quality maintained."
  • The cache matches near-literal repeats (exact or ≥0.85 token similarity). Real agents reword the same intent, so production needs embedding-based matching (cosine ≥ 0.95).
  • The kill thresholds are hardcoded (>3 identical calls, 800K tokens/run) and the buggy agent loops on an identical prompt — the easy case. Real runaways vary slightly, so you'd want fuzzy matching or cost-based circuit breakers.
  • The 31% cache rate is specific to this scripted ticket. Workflows with fewer repeats cache less.
  • State is in-memory today. Production swaps in Redis + Postgres, real provider calls (LiteLLM/Bedrock/OpenAI/Anthropic), and a learned router policy.

7 The roadmap — intelligent routing

The pitch: "Today we route by rules. The next version routes by evidence — every call teaches the router which model to pick next time."

Today — rules-based

static, hardcoded, no feedback
  • Task type maps to a fixed model tier.
  • codegen maps to frontier in the rule table. classify always → small.
  • Doesn't know about latency, live cost, or real quality.
  • Can't learn. Same decision on day 1 as day 365.

Next — data-driven

learns from every call, multi-provider
  • Routes based on historical performance per model per task type.
  • Knows live cost, latency, and measured quality — not estimates.
  • Multi-provider: OpenAI vs Anthropic vs Gemini vs Mistral.
  • Every call is feedback. Router improves over time.

Minimum viable prototype — 3 steps

StepWhat to buildWhy it matters
1 Instrument every call. Log task type, model, tokens, cost, latency, quality score to a Postgres table. This is the foundation. Nothing else works without this data. Observability IS the training data.
2 Replace hardcoded rules with a policy table. Router queries the DB: "for codegen tasks, which model has the best cost/quality ratio right now?" Now routing decisions are driven by evidence, not assumptions. Swap providers without touching code.
3 Add an LLM judge. After each response, a cheap eval model scores quality (0–1) and writes it back to the telemetry store. Closes the feedback loop. Router now has real quality signal — not modelled estimates.
